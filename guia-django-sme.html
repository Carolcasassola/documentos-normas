1 Princípios Gerais

  • Idioma único: pt-br. Modelos, campos, classes, funções, variáveis, rotas e mensagens são todos em português. Inglês é reservado apenas para termos técnicos consagrados (request, response, serializer, view, migration).
  • Separação de camadas explícita. Cada domínio é um app Django independente; dentro de cada app o fluxo é model → repository → service → serializer → view.
  • Configuração via ambiente. Todo valor que muda entre ambientes (chaves, URLs, portas, hosts, segredos) vem de variáveis de ambiente, nunca hard-coded.
  • Documentação como código. Docstrings obrigatórias, Sphinx para domínios, drf-spectacular para a API HTTP.
  • Lint/format/type-check automatizados. O que não passa no pre-commit não vai para o repositório.
  • Testes ao lado do código. Cada app guarda seus próprios testes.

2 Arquitetura de Pastas

Estrutura mínima de uma aplicação Django SME:

estrutura do repositório 
repo/
├── apps/                    # Apps Django (um por domínio de negócio)
│   ├── core/                # Autenticação, health, utilidades compartilhadas
│   └── <dominio>/           # Demais domínios (ex.: turmas, componentes_curriculares)
├── config/                  # Configurações Django (settings, urls, wsgi)
├── docs/                    # Documentação Sphinx (regras de domínio, decisões)
├── requirements/            # Dependências separadas por ambiente
├── scripts/                 # Scripts auxiliares
├── testes/                  # Testes E2E (Cypress, Postman) — não unitários
├── .env.example
├── .pre-commit-config.yaml
├── Dockerfile               # Produção
├── Dockerfile.dev           # Desenvolvimento
├── docker-compose.yml
├── docker-compose-dev.yml
├── manage.py
├── pyproject.toml
└── README.md

Pasta apps/

Todos os apps Django residem em apps/. Cada app segue a estrutura:

estrutura de cada app
apps/<dominio>/
├── __init__.py
├── apps.py
├── models.py
├── repository.py
├── services.py
├── serializers.py
├── constants.py
├── queries.py
├── api/
│   ├── __init__.py
│   ├── urls.py
│   └── views.py
└── tests/
    ├── __init__.py
    ├── test_views.py
    ├── test_services.py
    └── test_repository.py

Pasta config/

Substitui o pacote homônimo ao projeto criado pelo django-admin startproject. Centraliza configuração e roteamento:

  • config/settings.py — settings único, lendo do ambiente
  • config/urls.py — URL conf raiz
  • config/wsgi.py — entrada WSGI/ASGI
  • config/test_runner.py — runner de testes customizado (quando necessário)
🚫
O nome config é canônico no padrão SME. Não usar core_project, <nome_projeto> ou similares.

Pasta docs/

estrutura docs/
docs/
├── conf.py
├── index.rst
├── dominios/
│   └── <dominio>/
│       └── index.rst
└── make.bat / Makefile

4.1 Nomenclatura em Português

Modelos, campos, classes, métodos, variáveis e rotas em pt-br, sem acentos (compatibilidade com identificadores Python). Termos técnicos da stack permanecem em inglês (request, response, serializer, view, manager, queryset).

❌ Errado✅ Certo
EtlExecutionEtlExecucao
start_datedata_inicio
user_loginlogin_usuario
class StudentServiceclass AlunoService
/api/v1/classes//api/v1/turmas/

4.2 Configuração via .env

  • Todos os valores variáveis vivem em config/settings.py, lidos com os.environ e default seguro.
  • O arquivo .env real nunca é commitado; apenas .env.example documenta as variáveis suportadas.
  • Valores de exemplo no .env.example não devem expor portas reais, hostnames internos ou chaves de produção. Use placeholders descritivos (sua-chave-aqui, url_banco_dominio).
  • Variáveis sensíveis (senhas, tokens, secrets) devem ter default vazio ou genérico no settings.py.
  • A SECRET_KEY do Django sempre vem do ambiente — o default no código serve apenas para falhar visivelmente em produção quando não configurada.
🚫
Nunca usar valores como localhost:5432, redis://10.0.0.5 ou portas usadas em produção no .env.example. Isso evita reconhecimento de superfície interna em vazamentos do repositório.

4.3 Pasta de Settings renomeada para config/

O django-admin startproject cria um pacote com o mesmo nome do projeto. Após criar, renomeie para config/ e atualize:

manage.py
DJANGO_SETTINGS_MODULE = "config.settings"
config/settings.py
ROOT_URLCONF = "config.urls"
WSGI_APPLICATION = "config.wsgi.application"

Isso desacopla o nome do projeto da configuração e padroniza a importação entre microsserviços.

4.4 Camadas dentro de um App

O fluxo de uma requisição segue sempre a mesma cadeia:

🌐
HTTP
request
›
👁️
View
api/views.py
›
⚙️
Service
services.py
›
🗄️
Repository
repository.py
›
🏗️
Model
models.py
ArquivoResponsabilidade
models.pyDefine a estrutura de dados. Métodos só para coisas estritamente de modelo (__str__, propriedades derivadas).
repository.pyÚnica camada autorizada a executar queries ORM. Retorna dict ou tipos primitivos — não instâncias de modelo — para isolar o resto da aplicação do ORM.
services.pyOrquestra repositórios, aplica regras de negócio, combina dados. Não importa do DRF nem do Django HTTP.
serializers.pyValida entrada e formata saída. Sem regra de negócio.
api/views.pyFina: recebe request, delega ao service, devolve Response. Decorada com @extend_schema para documentação OpenAPI.
api/urls.pyApenas rotas do app, sem prefixo global (o prefixo vem do include() em config/urls.py).
constants.pyCódigos de domínio (tipos, modalidades, status) usados pelos demais módulos.

4.5 Autenticação

O esquema de autenticação é determinado pelo tipo da aplicação:

TipoEsquemaImplementação
MS Microsserviço API Key em header HTTP Segredo compartilhado entre serviços, configurado via .env (API_KEY, API_KEY_HEADER)
monolito JWT com integração ao CoreSSO Identidade do usuário emitida pelo CoreSSO (SSO institucional SME); o monolito valida o token e resolve perfis via API EOL

Regras comuns aos dois esquemas

  • Configurar em REST_FRAMEWORK["DEFAULT_AUTHENTICATION_CLASSES"], nunca por view.
  • Segredos (API Key, JWT signing key, credenciais do CoreSSO) sempre via .env.
  • Endpoint de health sempre público (AllowAny), nunca atrás de auth.
  • Documentar o esquema no schema OpenAPI via SPECTACULAR_SETTINGS["APPEND_COMPONENTS"]["securitySchemes"].

4.6 Documentação da API

Toda view deve ter:

  • Docstring de classe descrevendo a responsabilidade
  • Decorator @extend_schema(...) com tags, summary, parameters, request, responses e operation_id

5.1 Pre-commit

Arquivo .pre-commit-config.yaml na raiz, configurado com no mínimo:

  • black — formatação
  • ruff --fix — lint e ordenação de imports
  • mypy — tipagem estática
bash
# Instalação obrigatória após clonar o repositório
pre-commit install
🚫
O que não passa no pre-commit não vai para o repositório.

5.2 Docstrings — Padrão Google

Todas as funções, métodos e classes públicas têm docstring conforme o padrão Google:

python
def buscar_turma(codigo: int) -> dict | None:
    """Retorna dados cadastrais de uma turma.

    Args:
        codigo: Código identificador da turma no EOL.

    Returns:
        Dicionário com os dados da turma, ou None se não encontrada.

    Raises:
        TurmaInvalidaError: Se o código informado não for válido.
    """

Seções suportadas: Args, Returns, Raises, Yields, Example. A configuração das regras pydocstyle do Ruff no pyproject.toml deve habilitar as regras D para validação automática.

5.3 Sphinx para Documentação de Domínio

O foco do Sphinx não é documentar a API HTTP (responsabilidade do drf-spectacular) — é documentar:

📋 O que documentar no Sphinx
  • Regras de negócio do domínio
  • Decisões arquiteturais (ADRs leves)
  • Mapeamento com sistemas legados (ETL, EOL)
  • Glossário de termos do negócio
🔨 Build
bash
sphinx-build -b html \
  docs/ docs/_build/html

5.4 Requirements Separados

🏭 requirements/base.txt

Somente o que roda em produção.

💻 requirements/local.txt

Começa com -r base.txt e adiciona ferramentas de desenvolvimento, linters, coverage, pre-commit e debugpy.

🚫
Nunca colocar pytest, coverage, black, ruff, mypy ou pre-commit em base.txt: essas ferramentas pertencem ao ambiente de desenvolvimento.

5.5 Testes dentro de cada App

estrutura de testes por app
apps/<dominio>/tests/
├── __init__.py
├── test_views.py
├── test_services.py
└── test_repository.py
✅ Vantagens
  • Coesão — testes acompanham o código que verificam
  • Encapsulamento — mover/remover um app remove seus testes junto
  • Discoverability — quem abre o app vê o que está coberto
🗂️ Pasta testes/ (raiz)

Reservada para testes E2E / smoke com Cypress, Postman ou BDD. Testes unitários ficam dentro de cada app.

5.6–5.7 Cobertura e Banco em Testes

📊 Cobertura mínima

Mínimo recomendado de 80% por app, com prioridade para services.py e repository.py. Configurado no pyproject.toml.

80%
cobertura mínima
🗄️ Banco em testes

Usar SQLite em memória durante testes para velocidade e isolamento, mantendo PostgreSQL como banco de produção. A troca do ENGINE pode ser feita em config/settings.py quando o argumento test estiver presente em sys.argv.

5.8 Configuração do pyproject.toml

O pyproject.toml centraliza os metadados do projeto e as configurações das ferramentas de qualidade. Isso evita configurações espalhadas em arquivos diferentes.

pyproject.toml
[build-system]
requires = ["setuptools>=68", "wheel"]
build-backend = "setuptools.build_meta"

[project]
name = "<nome-do-projeto>"
version = "0.1.0"
description = "<descrição-do-projeto>"
readme = "README.md"
requires-python = ">=<versao-python>"
authors = [{ name = "SME" }]

[tool.setuptools.packages.find]
where = ["."]
include = ["apps*", "config*"]

[tool.black]
line-length = 79
target-version = ["<alvo-python>"]
exclude = "/migrations/"

[tool.ruff]
line-length = 79
target-version = "<alvo-python>"
exclude = ["migrations"]

[tool.ruff.lint]
select = ["E", "F", "W", "I", "N", "UP", "B", "C4", "SIM", "D"]

[tool.ruff.lint.per-file-ignores]
"**/__init__.py" = ["D104"]
"**/tests/**/*.py" = ["D100", "D101", "D102", "D103"]

[tool.ruff.lint.pydocstyle]
convention = "google"

[tool.mypy]
python_version = "<versao-python>"
exclude = ".*/migrations/.*"
warn_return_any = true
disallow_untyped_defs = true
ignore_missing_imports = true

[tool.pytest.ini_options]
DJANGO_SETTINGS_MODULE = "config.settings"
python_files = ["test_*.py"]
python_classes = ["Test*"]
python_functions = ["test_*"]

[tool.coverage.run]
source = ["apps"]

[tool.coverage.report]
show_missing = true
fail_under = 80
⚠️
Critérios de adaptação: Não adicionar ignore global para regras de docstring — exceções devem ser restritas com per-file-ignores. Não usar versões flutuantes: a versão estável mais recente deve ser adotada conscientemente na criação do projeto e deve permanecer fixada e consistente no pyproject.toml, Dockerfile e CI.

5.9 Docker

🏭 Dockerfile (produção)
  • Slim, instala apenas base.txt
  • Usa Gunicorn como servidor WSGI
  • Base: python:<versao>-slim
💻 Dockerfile.dev (desenvolvimento)
  • Instala local.txt + debugpy
  • Hot reload habilitado
  • Volume mount via docker-compose-dev.yml
🚫
Não usar:
  • Tags flutuantes como python:latest ou python:slim — podem mudar entre builds sem alteração no código.
  • Alpine — incompatibilidades recorrentes com Psycopg, gRPC para Python e wheels binários.

6 Checklist para Novos Projetos

Ao criar uma nova aplicação Django SME (monolito ou microsserviço), garantir antes do primeiro merge:

📁 Estrutura
Pasta apps/ com pelo menos apps/core/
Pasta config/ (renomeada do default do startproject)
Pasta requirements/ com base.txt e local.txt
Pasta docs/ com Sphinx configurado
Testes dentro de cada app, não na raiz
📄 Arquivos obrigatórios na raiz
pyproject.toml com black, ruff, mypy, coverage configurados
.pre-commit-config.yaml com black, ruff, mypy
.env.example sem valores reais nem portas explícitas
.gitignore cobrindo .env, __pycache__, .venv, .coverage
Dockerfile (prod) e Dockerfile.dev
docker-compose.yml e docker-compose-dev.yml
README.md com instruções de execução, endpoints e autenticação
💻 Código
Modelos, campos, classes e rotas em pt-br
Camadas separadas: model → repository → service → serializer → view
Docstrings Google em todas as classes/funções públicas
Tipagem estática (mypy disallow_untyped_defs)
Esquema de autenticação adequado: MS API Key ou monolito JWT + CoreSSO
Endpoint /api/v1/health/ público
Schema OpenAPI em /api/v1/schema/ e Swagger em /api/v1/docs/
🧪 Qualidade
Pre-commit instalado e passando (pre-commit install)
Cobertura mínima de 80%
