4 Modelos Arquiteturais Permitidos

4.1 Monólito Modular

✅ Permitido e Recomendado quando definido pela Arquitetura.

  • Separação clara por domínio
  • Services layer obrigatória
  • Proibição de dependência cruzada informal
  • Preparação para futura extração

4.2 Microserviços Orientados a Domínio

⚠️ Permitido com Justificativa Formal

  • Banco de dados isolado por serviço
  • Contrato versionado
  • Testes de contrato obrigatórios
  • Observabilidade implementada
  • Documentação C4 com boundary explícito

5 Diretriz para Adoção de Microserviços

ℹ️
Arquitetura distribuída não é objetivo em siA adoção de microserviços deve decorrer de necessidade estrutural comprovada — necessária em grandes sistemas e processamentos, visando escalabilidade horizontal e computação distribuída.
  • Coerência com o ecossistema existente
  • Capacidade operacional do time
  • Compatibilidade com legados
  • Sustentabilidade de longo prazo

6 Sistemas Estratégicos

Aplica-se a: NovoSGP Novo Pedagógico SME Identity

6.1 Fatiamento por Domínio

  • Corte orientado a domínio funcional
  • Boundary explícito obrigatório
  • Proibido fracionamento técnico artificial

6.2 Compatibilidade com Legados

  • Manutenção de contratos compatíveis
  • Versionamento controlado
  • Não interrupção de integrações existentes
  • Evolução incremental

6.3 Camada Anticorrupção

⛔ Obrigatória quando houver integração com modelo legado incompatível, exposição de ferramenta como endpoint, ETL, divergência semântica ou necessidade de isolamento.

A camada anticorrupção deve proteger o domínio interno e ser documentada no C4.

8 API-First e Contract-First

📄
Contrato é ativo institucional.APIs são definidas em conjunto com negócio, arquitetura e liderança técnica. Independente da modernização tecnológica, os contratos devem ser preservados.
  • Contrato construído colaborativamente
  • Responsáveis registrados formalmente
  • Versionamento obrigatório
  • Validação formal antes de publicação
  • Breaking change (major) exige nova versão obrigatoriamente

9 Versionamento Tecnológico

9.1 .NET

  • Sistemas novos → versão LTS mais recente aprovada institucionalmente
  • Sistemas legados (v2, v5 ou anteriores) → plano de progressão obrigatório

  • Refatoração incremental obrigatória
  • Proibido reescrita integral sem justificativa formal

13 Segurança

🔒
Responsabilidade do DesenvolvedorO desenvolvedor deve ser cuidadoso e responsável para que seu código esteja em conformidade com os padrões elevados de segurança da informação.
  • Observância ao OWASP Top 10
  • Autenticação centralizada
  • Validação server-side obrigatória
  • Logs estruturados
  • Mascaramento / criptografia de dados sensíveis
  • ⛔ Não commitar arquivos de variáveis de ambiente
  • ⛔ Não commitar arquivos lock
  • Auditoria de dependências via GitHub Dependabot
  • Política de atualização de bibliotecas críticas

14 Modelo de Evolução Tecnológica

14.1 Estratégia Geral

Evolução deve ser incremental, compatível e planejada. Toda decisão que impacte contrato público exige ADR aprovado por Arquitetura e validado por COTIC.

14.2 Roadmap .NET

  • LTS vigente para novos sistemas
  • Plano de migração para versões legadas
  • Compatibilidade de API durante upgrades
  • Evitar reescritas disruptivas
  • Clean Code · SOLID · Design Patterns GOF obrigatórios
  • Minimizar débitos técnicos sempre que possível

14.3 Roadmap Python

  • Versões oficialmente suportadas
  • Obedecer premissas das PEP e suas atualizações
  • Garantir compatibilidade de contrato
  • Clean Code · SOLID · Design Patterns GOF obrigatórios
  • Minimizar débitos técnicos sempre que possível

14.4 Modernização de Legados

✅

📖 Glossário

TermoDefinição
ArquiteturaÁrea que define padrões técnicos e estrutura dos sistemas.
COTICInstância que valida e prioriza decisões técnicas.
Contrato (API)Regras formais que definem como um sistema conversa com outro.
API-FirstPrimeiro define-se o contrato, depois implementa-se o sistema.
MicroserviçoSistema pequeno e independente, usado quando há necessidade real de escala ou isolamento.
Monólito ModularSistema único, mas organizado por partes bem separadas.
LegadoSistema já existente que precisa continuar funcionando.
VersionamentoControle de mudanças para não quebrar integrações existentes.
C4Modelo padrão de documentação arquitetural. c4model.com
ADRRegistro formal de decisão técnica (Architecture Decision Record).
